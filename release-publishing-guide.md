### Release Packaging & Publishing Setup Guide

This repository uses two workflows together:

- `General Build` builds/tests/publishes NuGet and creates tags.
- `Release Packaging & Publishing` starts from `General Build` success, reads its `release-metadata` artifact, creates release packages, creates/updates GitHub Release, and optionally publishes to external ecosystems.

### Required baseline configuration

1. Configure `.NET` version (no fallback is used):
   - Set `WORKFLOW_DOTNET_VERSION` in repository variables (`Settings -> Secrets and variables -> Actions -> Variables`) or in the repository `.env` consumed by workflows.
2. Configure NuGet push key for `General Build`:
   - Secret: `NUGET_KEY`
3. Ensure release trigger path is used:
   - Official release path: run `General Build` via `workflow_dispatch` on `main` (or with `push_packages == true` on `main`) so tags + metadata are produced.
   - PR distribution path: any `pull_request` run of `General Build` produces metadata and triggers distribution packaging.

### PR distribution trigger

- Trigger: `General Build` run with event `pull_request`
- Scope: any PR branch covered by `General Build` PR trigger configuration
- Effect:
  - Triggers package distribution flow from `Release Packaging & Publishing` via `workflow_run`.
  - Builds package artifacts and uploads them to the workflow artifacts.
  - Does **not** create/update GitHub Release and does **not** publish to external ecosystems (`WinGet`, `Chocolatey`, `AUR`, `Flathub`).

### Optional publish targets (simple setup only): secrets/variables and how to obtain them

#### WinGet

- Secret: `PERSONAL_ACCESS_TOKEN` (reused shared GitHub PAT)
- Optional override secret: `WINGET_TOKEN` (if you want dedicated token only for WinGet)
- Variable: `WINGET_PACKAGE_IDENTIFIER` (example: `Vendor.App`)
- How to get token:
  - Reuse the existing repository PAT (`PERSONAL_ACCESS_TOKEN`) with permissions needed to create PRs (typically `public_repo`) from the account used for WinGet submissions.
- Notes:
  - `winget-releaser` opens/updates manifests in `microsoft/winget-pkgs`.

#### Chocolatey

- Secret: `CHOCOLATEY_API_KEY`
- Optional variable: `CHOCOLATEY_PACKAGE_ID` (default: `editnet`)
- How to get key:
  - Sign in to Chocolatey.org -> user account -> API Keys -> create/copy API key.
- Script used by workflow:
  - `./scripts/publish-chocolatey.sh` (already included in this repository)

#### AUR

- Secret: `AUR_SSH_PRIVATE_KEY`
- Variable: `AUR_PACKAGE_NAME`
- Optional variable: `AUR_BINARY_NAME` (default: `edit-net`)
- How to get key:
  - Create SSH key pair dedicated to CI, add public key to AUR account, store private key in GitHub secret.
- Script used by workflow:
  - `./scripts/publish-aur.sh` (already included in this repository)

#### Flathub

- Secret: `PERSONAL_ACCESS_TOKEN` (reused shared GitHub PAT) or `FLATHUB_TOKEN`
- Variable: `FLATHUB_APP_ID`
- Optional variables:
  - `FLATHUB_MANIFEST_REPO` (default: `flathub/<FLATHUB_APP_ID>`)
  - `FLATHUB_MANIFEST_PATH` (default: `<FLATHUB_APP_ID>.yml`)
  - `FLATHUB_BRANCH` (default: `master`)
- How to get token:
  - Use a GitHub token that has push access to your Flathub manifest repository/fork used for updates. The workflow reuses `PERSONAL_ACCESS_TOKEN` by default and falls back to `FLATHUB_TOKEN` if provided.
- Script used by workflow:
  - `./scripts/publish-flathub.sh` (already included in this repository)

### What you need to do next (checklist)

1. Add `WORKFLOW_DOTNET_VERSION` variable.
2. Add `NUGET_KEY` secret.
3. For each external ecosystem you want to enable:
   - Add listed secrets + variables for supported simple targets only (`WinGet`, `Chocolatey`, `AUR`, `Flathub`).
   - Reuse `PERSONAL_ACCESS_TOKEN` for GitHub-based publishing (`WinGet`, `Flathub`) unless you explicitly provide dedicated override secrets.
   - Use built-in scripts in `./scripts/publish-*.sh` for `Chocolatey`, `AUR`, and `Flathub` (WinGet is action-based).
4. Run `General Build` (`workflow_dispatch`) on `main` with package push enabled.
5. Confirm `Release Packaging & Publishing` run:
   - Finds `release-metadata` artifact,
   - creates/updates GitHub Release with autogenerated notes,
   - uploads artifacts (`.AppImage`, `.deb`, `.rpm`, `.dmg`, `.msix`, `.exe`).
6. (Optional PR path) Open/update a PR and confirm distribution run uploads package artifacts.

### Excluded targets

- To keep setup minimal (key/token + optional package name/identifier), the workflow currently excludes URL/custom-endpoint style integrations such as `Homebrew`, `apt`, and `COPR`.
